# 要件定義書（Functional Specification）

このファイルは、開発の前提資料として一般的な、「要件定義書」です。  
下流の実態から想像して、サンプルとして用意したものです。

このフォルダについての説明は、[`README.md`](./README.md) に記載しています。

## 対応技術・前提環境

- HTML：HTML Living Standard
- CSS：FLOCSSを設計手法として使用
- JavaScript：動的表示に使用（一部を除き、HTML生成フェーズで処理）
- 開発支援：Node.js + Gulp によるビルド・管理
- 文書化：Markdownで記述し、後にHTML化

## 機能的要件

- ページ構成は静的HTML文書とし、スタイルは外部CSSで適用。
- スマホ向け表示にも対応するレスポンシブ設計。
- JavaScript に関する要件：
  - HTMLページ内で動的なインタラクションが必要な一部要素（例：ドロップダウンメニューの開閉、スムーズスクロール等）は、クライアントサイドのJavaScriptで制御する。
  - ただし、ナビゲーション構造やHTMLの主要構成要素は、開発時にNode.js上でJSONデータをもとに生成され、Pugにより静的にHTML化される。このため、JavaScriptはあくまで「表示後の補助的な動作」に限定される。
  - 外部スクリプトファイルとして提供されるJavaScriptは最小限とし、動作の分離性と管理のしやすさを確保する。
  - サイト構造情報（ナビゲーションの階層構造やパンくずリスト等）は、Node.jsスクリプトで自動生成され、Pugで処理されてHTML文書に含まれる。
  - 必要に応じて、構造データは `<script type="application/ld+json">` タグとして自動出力され、検索エンジン向けの構造化データにも対応する。
- ビルドツールとしてGulpを利用し、以下を自動化：
  - Sassのコンパイル
  - HTMLテンプレートの展開
  - ファイル構造の整理

## 非機能的要件

- 実装内に学習・設計の意図が明示されていること。
- Markdown文書は工程別に整理され、最終的にWebページとして出力可能。
- コードや文書が再利用可能な構造になっていること。

## 文書構成との関係

この要件定義に基づき、以下の中下流文書が連動して作成される：

- 構造設計書（HTML/CSSの階層やルール）
- 基本原則（設計・命名・管理方針）
- 実装規約（FLOCSSの運用規則など）
- 作業工程記録（プロセスの実践ログ）
